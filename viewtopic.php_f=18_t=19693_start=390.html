Sun Oct 26, 2003 4:33 pm by Chips

Trail.

Sometimes i can type for hours on how to do things, but for now it would take a long long time to explain how it works exactly. Here is a breif run down though!

The game runs off of a game engine. however, it gets the info to make things from data files - ie like stations, planets etc etc.

Now - these data files will contain stuff like references for certain other 'picture' files. They aren't really pictures, but its easy to call them that. Basically -there are several types - sur, mat, cmp and others too.
The sur files define the surface of things, which means you can 'collide' with them. The CMP files will define the points on the object where we can mount things from a basic prospective - but they contain alot more than that in reality. They also tell it where meshes, mats and other things will all go etc.

The mat file is the graphic part - like the colour etc etc.

Now - to reference all these to give an object an appearance, it is done in some files. For stations and planets - its in something called the solararch.ini files. This will correlate the data of where these MAT files, CMP files and SUR files are located to help the game find them quick.

This means that rather than reference those three/four files each time you use something in every system file for the game, you can enter just one line to reference it too a file that contains all the info necessary. It is just neater that way - and alot less complicated!.

That is what we do when adding objects to systems - like putting in new planets and bases etc. We just add a line that lets it know to look for the info prestated in another file - the solararch.ini file.

This means that adding planets and bases is as easy as pie. If it wasn't for this, it could be alot more complicated.

So - making new bases that way is easy, if you actually made brand new bases and planets - it would be so so much more difficult. for one, you would need all those new files - and as of yet we cannot even make sur files.

So - enough of that , where is the interesting stuff eh?

Okay - lets look

To add something to a file - we need to make sure we get the right file. For systems - its the system file FOR THAT SYSTEM. This is the fun bit.

Inside the data folder, there is a universe folder. Inside of their you will find systems folders - promising eh?
now - there are lots of folders - this is becuase they use nicknames.
So - Li is liberty
Ku is Kusari
Br is Bretonia
and Rh is Rheinland

now - you notice they have numbers too eh? That is becuase it makes it even easier as well. 01 is the first system (usually the capital system).
So -
Li01 is New York system
Li02 is Los Angeles system
Li03 is Colorado

Rh01 is New Berlin
Rh04 is Frankfurt

Br01 Is New london
Br04 Is Leeds

Ku01 is New tokyo
Ku04 is Honshu

Bw05 is Sigma 13

Hi01 is Omicron Alpha
Hi02 is Omicron Gamma

You see the nickname method? ALot shorter. Also - easy to cross reference for things like jumpgates (where from and too).

I suggest you get the Miscillaneous Freelancer Database by SuBXeRo from the downloads section. It is very helpful for this - it tells you what they all are!
Now - when you go to the li01 folder - it contains all the info for the New York system.
li01.ini - this file contains all the info about what is in new york system. everything!

Now then - use BiniQDU to decrypt the ini file into a text file (li10.ini.txt) so that you can 'read' it. Now - stuff saying [object -(Notice there SHOULD be a closing square bracket here), means that an object is described below it - like a station or planet etc.

So - here is manhatten in this:

[Object
nickname = Li01_01
ids_name = 196766
pos = -33270, 0, -33039
rotate = 0, -10, -10
ambient_color = 255, 0, 0
Archetype = planet_earthgrncld_4000
ids_info = 65759
spin = 0, 0.00100, 0
atmosphere_range = 4200
burn_color = 160, 222, 245
base = Li01_01_Base
reputation = li_p_grp
visit = 1

You notice the lines for this though? well - they are for different things. THe nickname is for the computer games use - it has NOTHING to do with the ingame name. That is the ids_name and ids_info entries. So the nickname is unique in each system. It just 'identifies' this object if it was referred too (hence it needs to be unique). The position defines where the heck in space it is - like the quadrant that is contained within.

Anyways - the reputation is for the group
